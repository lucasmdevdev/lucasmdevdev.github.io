How to Implement API Rate Limiting in Node.js (2026 Guide)

Rate limiting is the single most important defense your API has against abuse, DDoS attacks, and runaway costs. This guide covers everything from basic express-rate-limit to production-grade Redis-backed sliding window implementations.

If you expose an API to the internet without rate limiting, it is only a matter of time before something goes wrong. A single misbehaving client can bring down your entire service, rack up cloud bills, or scrape your data dry. Rate limiting solves this by capping how many requests a client can make in a given time window.

In this guide, we will build rate limiting from scratch in Node.js, starting with the simplest approach and progressively moving to production-grade solutions. Every code example is tested and ready to use.

Why Rate Limiting Matters in 2026

APIs are the backbone of modern applications. Whether you are building a public REST API, a microservice mesh, or a simple backend for your SaaS, rate limiting protects you on multiple fronts:

  • Preventing abuse — bots, scrapers, and malicious actors cannot overwhelm your server
  • Fair usage — no single client monopolizes your resources at the expense of others
  • Cost control — downstream services (databases, third-party APIs) have their own limits; rate limiting prevents cascade failures
  • Compliance — many SLAs and API partnerships require you to enforce rate limits
  • Stability — predictable load means predictable performance

The cost of not having rate limiting is almost always higher than implementing it. Let us start with the simplest approach.

1. Quick Start with express-rate-limit

The express-rate-limit package is the most popular rate limiting middleware for Express.js. It works out of the box with zero configuration for simple use cases.

Installation

npm install express express-rate-limit

Basic Setup

import express from 'express';
import rateLimit from 'express-rate-limit';

const app = express();

// Global rate limit: 100 requests per 15 minutes per IP
const globalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 100,                    // limit each IP to 100 requests per window
  standardHeaders: true,       // Return rate limit info in `RateLimit-*` headers
  legacyHeaders: false,        // Disable the `X-RateLimit-*` headers
  message: {
    status: 429,
    error: 'Too many requests. Please try again later.',
  },
});

app.use(globalLimiter);

app.get('/api/data', (req, res) => {
  res.json({ message: 'Here is your data' });
});

app.listen(3000, () => console.log('Server running on port 3000'));

This gives you immediate protection. Every IP address is limited to 100 requests per 15-minute window. When a client exceeds the limit, they receive a 429 Too Many Requests response.

Route-Specific Limits

Not all endpoints are equal. Your login endpoint should have stricter limits than your public data endpoint:

// Strict limit for authentication routes
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,  // only 5 login attempts per 15 minutes
  message: {
    status: 429,
    error: 'Too many login attempts. Please wait 15 minutes.',
  },
  skipSuccessfulRequests: true,  // don't count successful logins
});

// Relaxed limit for read-only endpoints
const readLimiter = rateLimit({
  windowMs: 1 * 60 * 1000,
  max: 60,  // 60 requests per minute
});

app.post('/api/auth/login', authLimiter, loginHandler);
app.get('/api/posts', readLimiter, getPostsHandler);
app.post('/api/posts', globalLimiter, createPostHandler);

The skipSuccessfulRequests option is particularly useful for login endpoints. If the user enters the correct password, it does not count against their limit. Only failed attempts accumulate.

2. Understanding Rate Limiting Algorithms

Before diving into more advanced implementations, it helps to understand the algorithms behind rate limiting. There are four main approaches:

Fixed Window

The simplest algorithm. Divide time into fixed windows (e.g., 1-minute blocks) and count requests in each window. When the count exceeds the limit, reject requests until the next window starts.

Pros: Simple, low memory usage. Cons: A burst of requests at the boundary of two windows can allow 2x the intended rate.

Sliding Window Log

Store a timestamp for every request. To check if a new request is allowed, count all timestamps within the last N seconds. This is perfectly accurate but uses more memory.

Sliding Window Counter

A hybrid approach. Keep counters for the current and previous fixed windows, then calculate a weighted count based on how far into the current window we are. This is what most production systems use.

// Sliding window counter formula:
// weighted_count = (previous_window_count * overlap_percentage) + current_window_count
//
// Example: window = 60s, we are 20s into the current window
// overlap = (60 - 20) / 60 = 0.667
// If previous window had 80 requests and current has 30:
// weighted_count = (80 * 0.667) + 30 = 83.3
// If limit is 100, this request is allowed

Token Bucket

Each client has a "bucket" of tokens that refills at a constant rate. Each request consumes one token. If the bucket is empty, the request is rejected. This naturally allows short bursts while maintaining an average rate.

3. Redis-Based Rate Limiting for Production

The in-memory store used by express-rate-limit has a critical limitation: it does not work across multiple server instances. If your API runs on 3 containers behind a load balancer, each container has its own counter. A client could make 100 requests to each container, effectively bypassing your limit.

Redis solves this by providing a shared, fast data store that all instances can access.

Using rate-limit-redis

npm install express-rate-limit rate-limit-redis ioredis
import express from 'express';
import rateLimit from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';
import Redis from 'ioredis';

const app = express();

const redisClient = new Redis({
  host: process.env.REDIS_HOST || '127.0.0.1',
  port: parseInt(process.env.REDIS_PORT || '6379'),
  password: process.env.REDIS_PASSWORD,
  enableOfflineQueue: false,  // fail fast if Redis is down
});

redisClient.on('error', (err) => {
  console.error('Redis connection error:', err);
});

const apiLimiter = rateLimit({
  windowMs: 60 * 1000,  // 1 minute
  max: 30,
  standardHeaders: true,
  legacyHeaders: false,
  store: new RedisStore({
    sendCommand: (...args) => redisClient.call(...args),
  }),
  message: {
    status: 429,
    error: 'Rate limit exceeded. Try again in a minute.',
    retryAfter: 60,
  },
});

app.use('/api/', apiLimiter);

app.get('/api/users', (req, res) => {
  res.json({ users: [] });
});

app.listen(3000);

Now every instance of your API shares the same rate limit counters through Redis. This is the standard approach for any production deployment.

4. Building a Sliding Window Rate Limiter from Scratch

Sometimes you need more control than a library provides. Here is a sliding window counter implementation using Redis sorted sets:

import Redis from 'ioredis';

const redis = new Redis();

interface RateLimitResult {
  allowed: boolean;
  remaining: number;
  resetAt: number;
  retryAfter: number | null;
}

async function slidingWindowRateLimit(
  key: string,
  limit: number,
  windowMs: number
): Promise<RateLimitResult> {
  const now = Date.now();
  const windowStart = now - windowMs;

  // Use a Redis pipeline for atomic operations
  const pipeline = redis.pipeline();

  // Remove expired entries
  pipeline.zremrangebyscore(key, 0, windowStart);

  // Count current entries
  pipeline.zcard(key);

  // Add the current request (we'll remove it if rejected)
  pipeline.zadd(key, now.toString(), `${now}-${Math.random()}`);

  // Set expiry on the key
  pipeline.pexpire(key, windowMs);

  const results = await pipeline.exec();
  const currentCount = results?.[1]?.[1] as number;

  if (currentCount >= limit) {
    // Over limit — remove the entry we just added
    await redis.zremrangebyscore(key, now, now);

    // Find when the oldest entry in the window expires
    const oldest = await redis.zrange(key, 0, 0, 'WITHSCORES');
    const oldestTime = oldest.length >= 2 ? parseInt(oldest[1]) : now;
    const retryAfter = Math.ceil((oldestTime + windowMs - now) / 1000);

    return {
      allowed: false,
      remaining: 0,
      resetAt: oldestTime + windowMs,
      retryAfter,
    };
  }

  return {
    allowed: true,
    remaining: limit - currentCount - 1,
    resetAt: now + windowMs,
    retryAfter: null,
  };
}

// Usage as Express middleware
function createSlidingWindowLimiter(limit: number, windowMs: number) {
  return async (req: any, res: any, next: any) => {
    const key = `ratelimit:${req.ip}`;
    const result = await slidingWindowRateLimit(key, limit, windowMs);

    // Set standard headers
    res.set('RateLimit-Limit', limit.toString());
    res.set('RateLimit-Remaining', result.remaining.toString());
    res.set('RateLimit-Reset', Math.ceil(result.resetAt / 1000).toString());

    if (!result.allowed) {
      res.set('Retry-After', result.retryAfter!.toString());
      return res.status(429).json({
        error: 'Too many requests',
        retryAfter: result.retryAfter,
      });
    }

    next();
  };
}

This implementation gives you precise sliding window behavior. The sorted set stores each request timestamp, and we remove expired entries on every check. The trade-off is slightly more Redis memory usage compared to a simple counter, but the accuracy is worth it for most APIs.

5. Token Bucket Implementation

The token bucket algorithm is ideal when you want to allow short bursts while maintaining an average rate. Here is a Redis-based implementation:

async function tokenBucket(
  key: string,
  bucketSize: number,      // max tokens (burst capacity)
  refillRate: number,       // tokens added per second
): Promise<{ allowed: boolean; tokens: number }> {
  const now = Date.now();
  const bucketKey = `bucket:${key}`;

  // Lua script for atomic token bucket operation
  const luaScript = `
    local bucket_key = KEYS[1]
    local bucket_size = tonumber(ARGV[1])
    local refill_rate = tonumber(ARGV[2])
    local now = tonumber(ARGV[3])

    local bucket = redis.call('HMGET', bucket_key, 'tokens', 'last_refill')
    local tokens = tonumber(bucket[1])
    local last_refill = tonumber(bucket[2])

    if tokens == nil then
      tokens = bucket_size
      last_refill = now
    end

    -- Calculate tokens to add since last refill
    local elapsed = (now - last_refill) / 1000
    tokens = math.min(bucket_size, tokens + (elapsed * refill_rate))
    last_refill = now

    local allowed = 0
    if tokens >= 1 then
      tokens = tokens - 1
      allowed = 1
    end

    redis.call('HMSET', bucket_key, 'tokens', tokens, 'last_refill', last_refill)
    redis.call('PEXPIRE', bucket_key, math.ceil(bucket_size / refill_rate) * 1000 + 1000)

    return {allowed, math.floor(tokens)}
  `;

  const result = await redis.eval(
    luaScript, 1, bucketKey,
    bucketSize, refillRate, now
  ) as [number, number];

  return {
    allowed: result[0] === 1,
    tokens: result[1],
  };
}

The Lua script ensures atomicity — no race conditions even under high concurrency. The bucket refills gradually, so a client that has been idle for a while gets their full burst capacity back.

6. Advanced Patterns

Per-User Rate Limiting with API Keys

IP-based rate limiting is a good default, but for authenticated APIs you should limit by user or API key:

const apiKeyLimiter = rateLimit({
  windowMs: 60 * 1000,
  max: 100,
  keyGenerator: (req) => {
    // Use API key if present, fall back to IP
    return req.headers['x-api-key']?.toString() || req.ip;
  },
  store: new RedisStore({
    sendCommand: (...args) => redisClient.call(...args),
  }),
});

Tiered Rate Limits

Different plans get different limits. This is how most SaaS APIs work:

function tieredLimiter() {
  const tiers = {
    free:       { windowMs: 60_000, max: 10 },
    starter:    { windowMs: 60_000, max: 60 },
    pro:        { windowMs: 60_000, max: 300 },
    enterprise: { windowMs: 60_000, max: 1000 },
  };

  return async (req: any, res: any, next: any) => {
    const user = req.user;  // from auth middleware
    const tier = user?.plan || 'free';
    const config = tiers[tier] || tiers.free;

    const key = `ratelimit:${tier}:${user?.id || req.ip}`;
    const result = await slidingWindowRateLimit(key, config.max, config.windowMs);

    res.set('RateLimit-Limit', config.max.toString());
    res.set('RateLimit-Remaining', result.remaining.toString());
    res.set('X-RateLimit-Plan', tier);

    if (!result.allowed) {
      return res.status(429).json({
        error: 'Rate limit exceeded',
        plan: tier,
        upgrade: tier === 'free' ? 'https://yourapi.com/pricing' : undefined,
      });
    }

    next();
  };
}

Graceful Degradation When Redis Is Down

Your API should not go down just because Redis is unavailable. Implement a fallback:

import rateLimit from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';

function createResilientLimiter(options: any) {
  let useRedis = true;

  const redisClient = new Redis({
    host: process.env.REDIS_HOST,
    enableOfflineQueue: false,
    retryStrategy: (times: number) => {
      if (times > 3) {
        useRedis = false;
        console.warn('Redis unavailable, falling back to in-memory rate limiting');
        return null;  // stop retrying
      }
      return Math.min(times * 200, 2000);
    },
  });

  redisClient.on('connect', () => {
    useRedis = true;
    console.log('Redis connected, using distributed rate limiting');
  });

  return rateLimit({
    ...options,
    store: useRedis
      ? new RedisStore({ sendCommand: (...args: any[]) => redisClient.call(...args) })
      : undefined,  // falls back to in-memory
  });
}

7. Rate Limit Response Headers

Good rate limiting is transparent. Your API should always tell clients about their limits through standard headers:

Header Purpose Example
RateLimit-Limit Maximum requests allowed in the window 100
RateLimit-Remaining Requests remaining in the current window 73
RateLimit-Reset Unix timestamp when the window resets 1711728000
Retry-After Seconds to wait before retrying (only on 429) 30

These headers follow the IETF RateLimit header fields draft standard. Clients can use them to implement automatic backoff and retry logic.

8. Testing Your Rate Limiter

Never deploy rate limiting without testing it. Here is a simple test script:

// test-rate-limit.ts
async function testRateLimit() {
  const url = 'http://localhost:3000/api/data';
  const results = { success: 0, limited: 0, errors: 0 };

  // Fire 150 requests rapidly
  const promises = Array.from({ length: 150 }, async (_, i) => {
    try {
      const res = await fetch(url);
      if (res.status === 200) results.success++;
      else if (res.status === 429) results.limited++;
      else results.errors++;

      // Log headers from first limited response
      if (res.status === 429 && results.limited === 1) {
        console.log('Rate limit headers:');
        console.log('  Limit:', res.headers.get('ratelimit-limit'));
        console.log('  Remaining:', res.headers.get('ratelimit-remaining'));
        console.log('  Reset:', res.headers.get('ratelimit-reset'));
        console.log('  Retry-After:', res.headers.get('retry-after'));
      }
    } catch {
      results.errors++;
    }
  });

  await Promise.all(promises);
  console.log(`\nResults: ${results.success} OK, ${results.limited} limited, ${results.errors} errors`);
}

testRateLimit();

9. Common Mistakes to Avoid

After building rate limiters for several production APIs, here are the pitfalls I see most often:

  1. Trusting X-Forwarded-For blindly — this header can be spoofed. Only trust it if your reverse proxy (nginx, CloudFlare) sets it and you strip client-provided values.
  2. Rate limiting after expensive work — check the rate limit before processing the request, not after. Put the middleware first.
  3. No fallback when Redis dies — always have a degradation strategy (see section 6).
  4. Same limits for all endpoints — your login route needs much stricter limits than your read endpoints.
  5. Forgetting WebSocket connections — rate limit connection attempts and messages separately.
  6. Not returning standard headers — clients need to know their limits to behave properly.

10. Complete Production Setup

Here is a complete, production-ready rate limiting setup combining everything we have covered:

// rate-limiter.ts
import rateLimit from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';
import Redis from 'ioredis';

const redis = new Redis({
  host: process.env.REDIS_HOST || '127.0.0.1',
  port: parseInt(process.env.REDIS_PORT || '6379'),
  password: process.env.REDIS_PASSWORD,
});

function createLimiter(name: string, windowMs: number, max: number) {
  return rateLimit({
    windowMs,
    max,
    standardHeaders: true,
    legacyHeaders: false,
    store: new RedisStore({
      sendCommand: (...args: any[]) => redis.call(...args),
      prefix: `rl:${name}:`,
    }),
    keyGenerator: (req) => req.user?.id || req.ip,
    handler: (req, res) => {
      res.status(429).json({
        error: 'Too many requests',
        limit: max,
        windowMs,
        retryAfter: res.getHeader('Retry-After'),
      });
    },
  });
}

export const globalLimiter = createLimiter('global', 60_000, 60);
export const authLimiter = createLimiter('auth', 15 * 60_000, 5);
export const uploadLimiter = createLimiter('upload', 60_000, 10);
export const searchLimiter = createLimiter('search', 60_000, 30);

// app.ts
import express from 'express';
import { globalLimiter, authLimiter, uploadLimiter, searchLimiter } from './rate-limiter';

const app = express();

app.use(globalLimiter);
app.post('/api/auth/*', authLimiter);
app.post('/api/upload', uploadLimiter);
app.get('/api/search', searchLimiter);

// ... your routes

Summary

Rate limiting is not optional for production APIs. Here is the decision tree:

  • Single server, low traffic — express-rate-limit with in-memory store is fine
  • Multiple instances or moderate traffic — add Redis with rate-limit-redis
  • High traffic with tiered plans — custom sliding window with Redis sorted sets
  • Need burst tolerance — token bucket algorithm

Start with express-rate-limit, add Redis when you scale, and customize when your requirements demand it. The important thing is to have rate limiting from day one.
