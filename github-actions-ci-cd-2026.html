GitHub Actions CI/CD: From Zero to Auto-Deploy (2026)

GitHub Actions is the most accessible CI/CD platform in 2026. Free for public repos, deeply integrated with GitHub, and powerful enough for any project. This guide takes you from zero to a fully automated test-build-deploy pipeline.

If you are still deploying by SSHing into a server and running git pull, this guide is for you. By the end, every push to your main branch will automatically run tests, build a Docker image, and deploy to production. No manual steps, no forgotten migrations, no "it works on my machine."

1. Your First Workflow

GitHub Actions workflows live in .github/workflows/ as YAML files. Let us start with the simplest possible workflow:

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

That is it. Push this file to your repository and GitHub will run your tests on every push to main and on every pull request. Let us understand each section:

  • on — defines when the workflow runs (triggers)
  • jobs — a workflow contains one or more jobs that run in parallel by default
  • runs-on — the virtual machine image to use
  • steps — sequential tasks within a job
  • uses — references a reusable action from the GitHub marketplace
  • run — executes a shell command

2. Running Tests with Services

Most real applications need a database to run tests. GitHub Actions supports service containers:

name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest

    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: testdb
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

      redis:
        image: redis:7
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - run: npm ci

      - name: Run database migrations
        run: npx prisma migrate deploy
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb

      - name: Run tests
        run: npm test
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb
          REDIS_URL: redis://localhost:6379
          NODE_ENV: test

      - name: Upload coverage
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: coverage/

The service containers start before your steps run and are automatically cleaned up afterward. Health checks ensure they are ready before your tests execute.

3. Linting and Code Quality

Add a separate job for linting. Since jobs run in parallel by default, this does not slow down your test suite:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check

  test:
    runs-on: ubuntu-latest
    # ... (test job from above)

Both lint and test run simultaneously. The workflow only passes if both succeed. This keeps your feedback loop fast.

4. Building and Pushing Docker Images

Once tests pass, build a Docker image and push it to a container registry:

  build:
    needs: [lint, test]  # Only run after lint and test pass
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # Only on main branch

    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha
            type=ref,event=branch
            type=semver,pattern={{version}}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

Key details:

  • needs: [lint, test] — this job waits for both lint and test to pass
  • cache-from/cache-to: type=gha — uses GitHub's built-in cache for Docker layers, dramatically speeding up subsequent builds
  • The metadata-action automatically generates useful tags based on the git ref
  • secrets.GITHUB_TOKEN is automatically available, no manual setup needed for GHCR

5. Secrets and Environment Variables

Never hardcode secrets in workflow files. GitHub provides encrypted secrets at the repository and organization level.

Setting Up Secrets

Go to your repository Settings, then Secrets and variables, then Actions. Add your secrets there. Common ones include:

  • DEPLOY_SSH_KEY — private SSH key for deployment
  • DEPLOY_HOST — server IP or hostname
  • DOCKER_REGISTRY_TOKEN — for private registries
  • SLACK_WEBHOOK — for deployment notifications

Using Secrets in Workflows

    steps:
      - name: Deploy to production
        env:
          SSH_KEY: ${{ secrets.DEPLOY_SSH_KEY }}
          DEPLOY_HOST: ${{ secrets.DEPLOY_HOST }}
        run: |
          echo "$SSH_KEY" > deploy_key
          chmod 600 deploy_key
          ssh -i deploy_key -o StrictHostKeyChecking=no \
            deploy@$DEPLOY_HOST \
            "cd /opt/app && docker compose pull && docker compose up -d"

Environments for Staged Deployments

GitHub Environments let you define different secrets and protection rules for staging vs production:

  deploy-staging:
    needs: build
    runs-on: ubuntu-latest
    environment: staging  # Uses staging secrets
    steps:
      - name: Deploy to staging
        run: ./deploy.sh staging

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://yourapp.com
    steps:
      - name: Deploy to production
        run: ./deploy.sh production

You can require manual approval before production deploys. Go to Settings, Environments, production, and add required reviewers. The workflow will pause and wait for approval before proceeding.

6. Matrix Builds: Testing Across Versions

Matrix builds let you test against multiple versions of Node.js, Python, operating systems, or any other variable:

  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node-version: [18, 20, 22]
      fail-fast: false  # Don't cancel other jobs if one fails

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test

This creates 9 jobs (3 OS x 3 Node versions) that run in parallel. fail-fast: false ensures all combinations are tested even if one fails — useful for identifying platform-specific issues.

Excluding and Including Specific Combinations

    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
        node-version: [18, 20, 22]
        exclude:
          - os: windows-latest
            node-version: 18  # Skip Node 18 on Windows
        include:
          - os: ubuntu-latest
            node-version: 22
            experimental: true  # Add a flag for special handling

7. Caching for Faster Builds

Caching dependencies is one of the most impactful optimizations. The actions/setup-node action handles npm caching automatically with the cache: 'npm' option. For other tools:

      - name: Cache Playwright browsers
        uses: actions/cache@v4
        with:
          path: ~/.cache/ms-playwright
          key: playwright-${{ runner.os }}-${{ hashFiles('package-lock.json') }}

      - name: Install Playwright
        run: npx playwright install --with-deps
        if: steps.cache.outputs.cache-hit != 'true'

The key includes a hash of the lock file, so the cache is invalidated when dependencies change. This can save 2-5 minutes per workflow run.

8. Complete CI/CD Pipeline

Here is a complete, production-ready pipeline combining everything we have covered:

# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run type-check

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: testdb
        ports: ['5432:5432']
        options: --health-cmd pg_isready --health-interval 10s --health-timeout 5s --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npx prisma migrate deploy
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb
      - run: npm test -- --coverage
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage
          path: coverage/

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    permissions:
      contents: read
      packages: write
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha
            type=raw,value=latest
      - uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://yourapp.com
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DEPLOY_HOST }}
          username: deploy
          key: ${{ secrets.DEPLOY_SSH_KEY }}
          script: |
            cd /opt/app
            docker compose pull
            docker compose up -d --remove-orphans
            docker image prune -f
            echo "Deployed at $(date)"

      - name: Verify deployment
        run: |
          sleep 10
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://yourapp.com/health)
          if [ "$STATUS" != "200" ]; then
            echo "Health check failed with status $STATUS"
            exit 1
          fi
          echo "Deployment verified successfully"

      - name: Notify on Slack
        if: always()
        uses: 8398a7/action-slack@v3
        with:
          status: ${{ job.status }}
          fields: repo,message,commit,author
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}

Key features of this pipeline:

  • concurrency — cancels in-progress runs when a new push arrives, saving runner minutes
  • Lint and test run in parallel for speed
  • Build only runs on pushes to main (not on PRs)
  • Deploy requires manual approval via the production environment
  • Health check verifies the deployment actually works
  • Slack notification on success or failure

9. Reusable Workflows

If you have multiple repositories with similar pipelines, extract common workflows:

# .github/workflows/reusable-test.yml
name: Reusable Test Workflow

on:
  workflow_call:
    inputs:
      node-version:
        required: false
        type: string
        default: '20'

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test
# .github/workflows/ci.yml — In any repository
name: CI
on: [push, pull_request]

jobs:
  test:
    uses: yourorg/.github/.github/workflows/reusable-test.yml@main
    with:
      node-version: '20'

This keeps your CI/CD DRY across dozens of repositories.

10. Useful Workflow Recipes

Auto-Label Pull Requests

name: Label PRs
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
    steps:
      - uses: actions/labeler@v5
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}

Scheduled Security Audits

name: Security Audit
on:
  schedule:
    - cron: '0 8 * * 1'  # Every Monday at 8 AM

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high
      - run: npx snyk test
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

Auto-Merge Dependabot PRs

name: Auto-Merge Dependabot
on: pull_request

permissions:
  contents: write
  pull-requests: write

jobs:
  auto-merge:
    if: github.actor == 'dependabot[bot]'
    runs-on: ubuntu-latest
    steps:
      - uses: dependabot/fetch-metadata@v2
        id: metadata
      - if: steps.metadata.outputs.update-type == 'version-update:semver-patch'
        run: gh pr merge --auto --squash "$PR_URL"
        env:
          PR_URL: ${{ github.event.pull_request.html_url }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

Summary

Here is the progression for any project:

  1. Week 1 — Add a basic CI workflow that runs tests on every push
  2. Week 2 — Add linting and type checking as a parallel job
  3. Week 3 — Add Docker image building with GHCR
  4. Week 4 — Add automated deployment with environment protection
  5. Ongoing — Add matrix builds, security audits, Dependabot auto-merge

Start simple. A 10-line workflow that runs your tests is infinitely better than no CI at all. You can always add complexity later.
