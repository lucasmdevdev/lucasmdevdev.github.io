Docker Compose for Production: The Complete Guide (2026)

Docker Compose is not just for local development. With the right configuration, it can power production workloads reliably. This guide covers everything you need to go from a dev-only docker-compose.yml to a battle-tested production setup.

Most Docker Compose tutorials stop at docker compose up and call it a day. That works fine on your laptop, but production is a different world. Services crash, disks fill up, secrets leak, and logs disappear. This guide addresses all of it.

We will build a complete production setup step by step: a Node.js API, PostgreSQL database, Redis cache, and Nginx reverse proxy. Every configuration decision is explained so you can adapt it to your own stack.

1. Production vs Development: What Changes

Before writing any YAML, let us be clear about what production requires that development does not:

Concern Development Production
Restart policy None or no unless-stopped or always
Health checks Optional Required
Secrets In .env file Docker secrets or external vault
Volumes Bind mounts for hot reload Named volumes with backups
Logging Console output Log drivers with rotation
Resources Unlimited CPU/memory limits set
Networking Default bridge Custom networks with isolation
Images Build locally Pull from registry with pinned tags

2. The Base: Multi-Service docker-compose.yml

Here is our complete production docker-compose.yml. We will break down every section afterward:

# docker-compose.yml — Production
version: "3.9"

services:
  # ---- API ----
  api:
    image: ghcr.io/yourorg/api:${APP_VERSION:-latest}
    restart: unless-stopped
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      NODE_ENV: production
      DATABASE_URL: postgresql://app:${DB_PASSWORD}@postgres:5432/appdb
      REDIS_URL: redis://redis:6379
      PORT: "3000"
    ports:
      - "127.0.0.1:3000:3000"  # Only expose to localhost
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 10s
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 128M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    networks:
      - frontend
      - backend

  # ---- PostgreSQL ----
  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d appdb"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 1G
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    networks:
      - backend

  # ---- Redis ----
  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 300M
    logging:
      driver: json-file
      options:
        max-size: "5m"
        max-file: "3"
    networks:
      - backend

  # ---- Nginx Reverse Proxy ----
  nginx:
    image: nginx:alpine
    restart: unless-stopped
    depends_on:
      api:
        condition: service_healthy
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf:ro
      - ./nginx/ssl:/etc/nginx/ssl:ro
      - nginx_cache:/var/cache/nginx
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 30s
      timeout: 3s
      retries: 3
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 128M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "5"
    networks:
      - frontend

volumes:
  postgres_data:
    driver: local
  redis_data:
    driver: local
  nginx_cache:
    driver: local

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true  # No external access

Let us break down the key decisions in this configuration.

3. Health Checks: The Foundation of Reliability

Health checks are the single most important production configuration. Without them, Docker has no way of knowing whether your service is actually working. A container can be "running" while the application inside it has crashed, deadlocked, or lost its database connection.

healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
  interval: 30s     # Check every 30 seconds
  timeout: 5s       # Fail if no response in 5 seconds
  retries: 3        # Mark unhealthy after 3 consecutive failures
  start_period: 10s # Grace period for startup

The health check endpoint itself should verify actual functionality, not just return 200:

// health.ts — Express health check endpoint
app.get('/health', async (req, res) => {
  const checks = {
    uptime: process.uptime(),
    timestamp: Date.now(),
    database: 'unknown',
    redis: 'unknown',
  };

  try {
    // Check database connection
    await prisma.$queryRaw`SELECT 1`;
    checks.database = 'healthy';
  } catch (err) {
    checks.database = 'unhealthy';
    return res.status(503).json(checks);
  }

  try {
    // Check Redis connection
    await redis.ping();
    checks.redis = 'healthy';
  } catch (err) {
    checks.redis = 'unhealthy';
    return res.status(503).json(checks);
  }

  res.json(checks);
});

With depends_on using condition: service_healthy, Docker Compose will not start your API until the database and Redis are actually ready. No more "connection refused" errors on startup.

4. Restart Policies: Surviving Crashes

There are four restart policies available:

  • no — Never restart (default). Good for development and one-off tasks.
  • always — Always restart, even if stopped manually. The container will also restart when the Docker daemon starts.
  • unless-stopped — Like always, but does not restart if the container was manually stopped. This is what you want for most production services.
  • on-failure — Only restart if the container exits with a non-zero code. Good for batch jobs.

Use unless-stopped for all long-running services. This means your services survive server reboots, OOM kills, and application crashes while still respecting manual docker compose stop commands.

5. Secrets Management

Never put secrets directly in your docker-compose.yml. Here are three approaches, from simplest to most secure:

Level 1: Environment File (Minimum Viable)

# .env (never commit this file)
DB_PASSWORD=super-secret-password-here
APP_VERSION=1.2.3
JWT_SECRET=another-secret
# docker-compose.yml
services:
  api:
    env_file:
      - .env

Make sure .env is in your .gitignore and .dockerignore. This is acceptable for small deployments but the secrets are visible in docker inspect.

Level 2: Docker Secrets

# Create secrets from files
echo "super-secret-password" | docker secret create db_password -

# docker-compose.yml
services:
  api:
    secrets:
      - db_password
    environment:
      DATABASE_URL: postgresql://app:FILE:/run/secrets/db_password@postgres:5432/appdb

secrets:
  db_password:
    external: true

Docker secrets are stored encrypted and only mounted into the containers that need them. They are available as files at /run/secrets/<name>.

Level 3: External Vault

For larger deployments, use HashiCorp Vault, AWS Secrets Manager, or similar. Your application fetches secrets at startup rather than receiving them through environment variables.

6. Network Isolation

The network configuration in our example is deliberate:

networks:
  frontend:
    driver: bridge
  backend:
    driver: bridge
    internal: true  # Key: no external access

The backend network is marked as internal: true. This means containers on this network cannot reach the internet. Your database and Redis are completely isolated — they can only communicate with other containers on the same network.

The API service is on both networks: it can talk to the database (backend) and receive requests from Nginx (frontend). Nginx is only on the frontend network. This is defense in depth.

7. Resource Limits: Preventing Runaway Containers

Without resource limits, a memory leak in your API can consume all available RAM and crash every other service on the host:

deploy:
  resources:
    limits:
      cpus: "1.0"      # Max 1 CPU core
      memory: 512M      # Max 512 MB RAM
    reservations:
      cpus: "0.25"      # Guaranteed 0.25 CPU
      memory: 128M      # Guaranteed 128 MB RAM

Set limits based on your actual usage patterns. Run your services under load and monitor with docker stats to find the right values. Start generous and tighten over time.

When a container exceeds its memory limit, Docker kills it with an OOM signal. Combined with restart: unless-stopped, it will come back immediately — containing the blast radius of the leak.

8. Logging Configuration

Docker's default logging keeps growing until your disk is full. Always configure log rotation:

logging:
  driver: json-file
  options:
    max-size: "10m"   # Rotate after 10 MB
    max-file: "3"     # Keep 3 rotated files

This means each service uses at most 30 MB of disk for logs. For production systems processing high volumes, consider shipping logs to an external service:

# Ship logs to a remote syslog server
logging:
  driver: syslog
  options:
    syslog-address: "tcp://logserver:514"
    tag: "{{.Name}}"

Or use a logging stack like Loki + Grafana for searchable, centralized logs. The Developer Starter Kit includes a pre-configured Loki setup you can drop into any Compose project.

9. Backup Strategy for Volumes

Named volumes persist your data, but they are not backed up by default. Here is a simple backup script:

#!/bin/bash
# backup.sh — Run daily via cron
set -euo pipefail

BACKUP_DIR="/backups/$(date +%Y-%m-%d)"
mkdir -p "$BACKUP_DIR"

# Backup PostgreSQL
docker compose exec -T postgres \
  pg_dump -U app -d appdb \
  | gzip > "$BACKUP_DIR/postgres.sql.gz"

# Backup Redis
docker compose exec -T redis \
  redis-cli BGSAVE
sleep 2
docker cp "$(docker compose ps -q redis)":/data/dump.rdb \
  "$BACKUP_DIR/redis.rdb"

# Clean backups older than 30 days
find /backups -maxdepth 1 -type d -mtime +30 -exec rm -rf {} +

echo "Backup completed: $BACKUP_DIR"

Add this to your crontab: 0 3 * * * /opt/app/backup.sh >> /var/log/backup.log 2>&1

10. Using Multiple Compose Files

Keep your base configuration in docker-compose.yml and override for different environments:

# docker-compose.override.yml — Development overrides (loaded automatically)
services:
  api:
    build: .
    volumes:
      - ./src:/app/src  # Hot reload
    environment:
      NODE_ENV: development
    ports:
      - "3000:3000"  # Expose directly
    restart: "no"

  postgres:
    ports:
      - "5432:5432"  # Direct access for dev tools
# docker-compose.prod.yml — Production overrides
services:
  api:
    image: ghcr.io/yourorg/api:${APP_VERSION}
    restart: unless-stopped
    # No volume mounts, no exposed ports except through nginx

Deploy with: docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

11. Zero-Downtime Deployments

For updates without downtime, use a rolling update strategy:

#!/bin/bash
# deploy.sh — Zero-downtime deployment
set -euo pipefail

NEW_VERSION=$1

echo "Pulling new image version: $NEW_VERSION"
APP_VERSION=$NEW_VERSION docker compose pull api

echo "Starting new container alongside old one"
APP_VERSION=$NEW_VERSION docker compose up -d --no-deps --scale api=2 api

echo "Waiting for new container to be healthy..."
sleep 15

echo "Removing old container"
APP_VERSION=$NEW_VERSION docker compose up -d --no-deps --scale api=1 api

echo "Deployment complete: $NEW_VERSION"

This starts a new container, waits for it to pass health checks, then removes the old one. Combined with an Nginx upstream configuration, clients experience zero downtime.

12. Monitoring with docker compose

Add a monitoring stack directly in your Compose file:

  # ---- Monitoring ----
  prometheus:
    image: prom/prometheus:latest
    restart: unless-stopped
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - prometheus_data:/prometheus
    networks:
      - backend

  grafana:
    image: grafana/grafana:latest
    restart: unless-stopped
    depends_on:
      - prometheus
    environment:
      GF_SECURITY_ADMIN_PASSWORD: ${GRAFANA_PASSWORD}
    volumes:
      - grafana_data:/var/lib/grafana
    ports:
      - "127.0.0.1:3001:3000"
    networks:
      - backend

This gives you metrics dashboards without any external dependencies. Monitor container health, resource usage, request rates, and error rates all in one place.

Production Checklist

Before deploying, verify every item:

  • All services have restart: unless-stopped
  • All services have health checks
  • All services have resource limits
  • All services have log rotation configured
  • Secrets are not hardcoded in YAML or Dockerfiles
  • Database ports are not exposed to the host
  • The backend network is internal: true
  • Named volumes are used (not bind mounts) for persistent data
  • Backup script is running on a schedule
  • Images use specific version tags, not latest
  • .env is in .gitignore and .dockerignore
